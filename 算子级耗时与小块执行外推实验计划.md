# 算子级耗时与小块执行外推实验计划

## 当前进展

- `cost_model_log_20260611_profile` 的算子级准确率已经达到 92.47%（MAPE 7.53%），当前 cost 估计本身已经可用。
- `analyze_chunk_actual_runtime_gap.py` 已证明，同一条 pipeline 的 chunk 实测耗时求和通常显著大于完整执行耗时，尤其是 image/text，说明“小块求和”不能直接等价于完整执行。
- 下一步重点不再是继续追 pipeline 级准确率，而是解释 chunk 场景下的固定开销与规模项，判断是否可用 `fixed_overhead + variable_cost_per_record * record_count` 近似完整执行耗时。

## 1. 实验背景与目标

当前阶段的工作重点从 pipeline 级准确率转向算子级代价估计与基数估计。本实验主要为其提供可靠的算子级 cost/cardinality 输入，并验证“预测 chunk 后再求和”这种低成本代价估计方式是否能够近似完整大数据集的真实执行耗时。

本轮耗时统计实验关注以下问题：

- 当前算子级 `op_process_time` 预测是否足够稳定。
- 当前算子级 `ds_output_count` 预测是否足够稳定。
- 对同一条 pipeline，将完整数据切成 chunk 后分别构造特征并使用代价模型预测，再把 chunk 级预测耗时求和，能否接近完整数据集真实执行耗时。
- 不同 chunk size 下，预测 chunk 求和相对完整执行的误差是否可接受。
- 如果预测 chunk 求和误差较大，需要分析问题来自模型预测误差、基数传播误差，还是 chunk 粒度过小导致特征表达和完整数据执行场景不一致。

## 4. Chunk 设置

第一轮建议使用以下 chunk size：

```text
100 records/files per chunk
200 records/files per chunk
500 records/files per chunk
1000 records/files per chunk
```

对应到不同模态：

```text
image：100 / 200 / 500 / 1000 张图一块
audio：100 / 200 / 500 / 1000 个音频文件一块
text：100 / 200 / 500 / 1000 条文本记录一块
```

chunk 实验应从 full baseline index 中切出互不重叠的小块 index 文件。不要直接用现有的 `128M/256M/512M` 或 `c4_1000/c4_2000/...` 这类 cumulative 子集替代 chunk 求和实验，除非后续确认这些文件本身就是互不重叠分片。

## 5. 如何比较完整数据和预测 chunk

比较单位是算子级，而不是 pipeline 级总准确率。

对同一条 pipeline，完整执行得到：

```text
operator_index = 1 的算子耗时 T_full_op1
operator_index = 2 的算子耗时 T_full_op2
...
```

对同一条 pipeline，将 full baseline index 切成多个 chunk。每个 chunk 构造算子级特征后送入当前 cost 模型，得到 chunk 级预测耗时：

```text
chunk_1 中 operator_index = 1 的预测耗时 T_chunk_pred_op1_1
chunk_2 中 operator_index = 1 的预测耗时 T_chunk_pred_op1_2
...

所有 chunk 中 operator_index = 1 的预测耗时求和 T_chunk_pred_sum_op1
所有 chunk 中 operator_index = 2 的预测耗时求和 T_chunk_pred_sum_op2
...
```

核心指标：

```text
chunk_pred_sum_relative_error
= |T_chunk_pred_sum_op - T_full_op| / T_full_op

chunk_pred_bias_ratio
= (T_chunk_pred_sum_op - T_full_op) / T_full_op
```

解释：

- `chunk_pred_bias_ratio > 0`：预测 chunk 求和高于完整执行真实耗时，说明该 chunk size 下预测偏保守或存在重复计入固定成本的问题。
- `chunk_pred_bias_ratio ≈ 0`：该 chunk size 下，预测 chunk 求和能够较好近似完整数据真实执行。
- `chunk_pred_bias_ratio < 0`：预测 chunk 求和低于完整执行真实耗时，说明模型可能低估了大规模执行中的 I/O、资源竞争、批处理效率变化等开销。

因此，8G 和预测 chunk 的区别需要逐算子对齐比较，但不是人工逐个分析，而是通过脚本自动按以下键生成统计表：

```text
ds_type + pipeline_name + chunk_size + operator_index + operator_name
```

然后再汇总到模态、算子类型和 chunk size。

## 6. 与当前模型预测结果的关系

完整数据执行结果用于得到真实算子耗时：

```text
op_process_time_full
```

chunk 数据不以真实执行求和作为主指标，而是用于构造多组较小输入规模下的算子级特征，并由当前 cost 模型预测：

```text
op_process_time_pred_chunk_sum
= sum(predicted chunk op time)
```

完整数据也可以直接构造特征并由当前 cost 模型预测：

```text
op_process_time_pred_full
```

主比较关系：

```text
chunk_prediction_relative_error
= |op_process_time_pred_chunk_sum - op_process_time_full| / op_process_time_full

full_prediction_relative_error
= |op_process_time_pred_full - op_process_time_full| / op_process_time_full
```

这样可以比较两种代价估计方式：

```text
方式一：直接预测完整数据特征 -> op_process_time_pred_full
方式二：预测多个 chunk 后求和 -> op_process_time_pred_chunk_sum
```

如果 `op_process_time_pred_chunk_sum` 接近 `op_process_time_full`，说明预测 chunk 后求和的开销估计方式可接受。若它比直接预测完整数据更稳定，则后续面对新 pipeline 或新数据集时，可以考虑先构造或采样小块特征，再辅助估计完整规模下的算子代价。

注意：这里的“预测时间”指模型预测出的算子执行耗时 `op_process_time_pred`，不是 AutoGluon 模型自身推理所消耗的时间。模型预测时应加载一次模型后批量预测所有 chunk，不要每个 chunk 重新启动预测脚本，否则会把脚本启动和模型加载开销混入实验。

## 8. 结果分析重点

实验最终需要回答：

- `100` 一块是否太小，导致预测 chunk 求和与完整执行真实耗时偏差较大。
- `200/500/1000` 中哪个 chunk size 的预测求和最稳定。
- 哪些算子的 chunk 级预测求和最接近完整数据真实执行。
- 哪些算子在 chunk 级预测中容易重复计入固定成本，或低估大规模执行中的 I/O、缓存、资源竞争等开销。
- 直接预测完整数据特征与预测 chunk 后求和相比，哪种方式更稳定。
- 后续算子重排优化时，是否可以用预测 chunk 求和作为完整数据集算子代价的低成本估计方式。

## 9. 检查项

数据完整性检查：

- full 真实执行结果应包含 `pipeline_name / operator_index / operator_name / ds_type / op_process_time / ds_input_count / ds_output_count`。
- chunk 特征和预测结果应包含 `pipeline_name / chunk_size / chunk_id / operator_index / operator_name / ds_type / op_process_time_pred / ds_input_count / ds_output_count_pred`。
- 每条 full pipeline 都应在 chunk 结果中找到同配置 pipeline。
- 同一 pipeline 下 full 与 chunk 的算子序列应一致。
- chunk 总输入样本量应接近 full baseline 的输入样本量，避免 chunk 数据漏跑。

指标正确性检查：

- 对一条 pipeline 手工抽查 1 个算子，验证 `op_process_time_pred_chunk_sum = 所有 chunk 中该算子预测耗时求和`。
- 验证 `chunk_pred_bias_ratio` 的正负含义。
- 验证结果表是否能按 `ds_type / operator_name / chunk_size` 正常聚合。

## 10. Assumptions

- 文档保存到项目根目录。
- 当前阶段不实现算子重排优化，只提供算子级 cost/cardinality 和耗时缩放分析依据。
- 图像和音频使用 8G 级 baseline。
- 文本使用当前最大可用 `c4_8000.jsonl` 作为 baseline，并在报告中单独说明其规模口径。
- 当前 cost/cardinality 的 MAPE 仍作为算子级模型基础准确率指标；UDF 算子执行效率提升率和资源利用率提升率属于后续算子重排优化模块的系统指标。
