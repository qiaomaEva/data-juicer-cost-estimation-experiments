# 数据采集实验

## **实验环境**

你的实验是基于 Docker 容器跑的 Data-Juicer 多模态批处理环境。核心容器是 yyw_dj_test，镜像是 xy_ray_env:v1。镜像历史显示它是基于 Ubuntu 22.04 构建的，内置 /home/ray/anaconda3/bin 这套 Python 环境，并安装了 Ray 2.51.1 相关 wheel。容器实际运行时用的是 root，工作目录是 /home/yyw/data-juicer，代码、数据和模型都通过挂载目录接入。

**资源与挂载**
容器申请了 GPU 能力，能看到整张卡；你们实际用的是 RTX A6000 48GB 这类环境。容器本身没有设 CPU/内存硬限制，但 shm-size 只有 64MB。3 个关键挂载目录是：

- /home/yyw/data-juicer：Data-Juicer 代码

## **离线依赖策略**

你现在的环境已经整理成“离线优先”模式，关键变量是：

- NLTK_DATA=/data/data-juicer_models/nltk_data
- HF_HOME=/data/data-juicer_models
- HF_HUB_CACHE=/data/data-juicer_models/hub
- HUGGINGFACE_HUB_CACHE=/data/data-juicer_models/hub
- TRANSFORMERS_CACHE=/data/data-juicer_models/hub
- HF_HUB_OFFLINE=1
- TRANSFORMERS_OFFLINE=1
- HF_DATASETS_OFFLINE=1
- PYTORCH_ALLOC_CONF=expandable_segments:True,max_split_size_mb:128

模型依赖：
文本:/data/data-juicer_models 
文本侧已经补齐并验证过的关键依赖包括 punkt、EleutherAI/pythia-14m、EleutherAI/pythia-6.9b-deduped。
图像:/data/data-juicer_dataset/coco2017/weights
图像侧显式依赖包括 NSFW、水印和 aesthetics 模型，隐式还会用到 YOLO/FastSAM 这类视觉模型。
音频:音频侧目前主要是 ffmpeg 和信号处理链，没看到像文本/图像那样重的 HF 模型依赖。

## **Pipeline 组织方式**

现在的实验结构分两层：

- 模板 YAML：audio_pipeline_*.yaml、image_pipeline_*.yaml、text_pipeline_*.yaml
- 展开后的批跑目录：prepared_ai/configs + prepared_ai/manifests

旧批次是 50 个文本、40 个图像、10 个音频模板。新批次是 50 个文本、40 个图像、30 个音频模板。现在已经有多尺度展开脚本 prepare_pipeline_runs_multiscale.py，它会按原实验风格生成：

- 音频：_128MB ... _8G
- 图像：_128MB ... _8G
- 文本：_1000/_2000/_5000/_8000/_10000

批量执行用的是 runner 脚本，结果输出你现在希望统一落到：

- runs/<run_id>/log
- runs/<run_id>/monitor
- runs/<run_id>/trace
- runs/<run_id>/<run_id>.jsonl

## **多模态算子与模型画像**

文本高频算子包括清洗、分句、分块、语言识别、统计过滤和去重，例如 sentence_split_mapper、text_chunk_mapper、alphanumeric_filter、token_num_filter、document_minhash_deduplicator 等。
图像高频算子包括 image_size_filter、image_shape_filter、image_aspect_ratio_filter、image_tagging_mapper、image_segment_mapper、image_remove_background_mapper、image_watermark_filter、image_nsfw_filter。
音频高频算子包括 audio_size_filter、audio_duration_filter、audio_nmf_snr_filter、audio_ffmpeg_wrapped_mapper、audio_add_gaussian_noise_mapper。

## **目前已知风险**

这套环境最需要注意的不是“缺依赖”，而是两类运行时问题：

- open_tracer=true 时，text_chunk_mapper、image_segment_mapper 这类会改变样本关系的算子容易触发 tracer 的长度断言
- 图像任务最容易受共享 GPU 干扰，尤其别人占着显存时，image_segment_mapper 会 OOM

另外文本里 document_minhash_deduplicator 已经出现过少量样本级空数组报错，本质是上游 chunk 后出现极短文本，进入 minhash 时没有有效 token，但这类通常不会让整任务退出。
