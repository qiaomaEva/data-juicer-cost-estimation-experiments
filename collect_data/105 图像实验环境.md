## 3. 数据资产清单

### 3.1 纯净原始图像数据 (`coco2017/train2017/`)

- **内容：** COCO 2017 训练集原始图像。
- **规模：** 完整包含 **118,288** 张纯净原图，占用存储空间约 **19.3 GB**。
- **⚠️ 极高危注意项（交接必读）：**
  - 该目录下**潜伏着 Data-Juicer 算子之前运行产生的历史缓存垃圾**。具体路径为 `train2017/__dj__produced_data__/`。
  - 这是之前在此服务器上运行 `image_face_blur_mapper`（脸部模糊算子）等实验时，系统隐性生成的一对多缓存废料，**文件数量高达 360 万+**。
  - **后续操作规范：** 任何人在对该 `train2017` 目录进行 `rsync` 同步、迁移或打包备份时，**必须**带上排除参数 `--exclude='__dj__produced_data__'`，否则会导致目标硬盘瞬间被废料塞满且极度耗时。

### 3.2 标准标注文件 (`coco2017/annotations/`)

- **内容：** 包含 COCO 官方标准的 JSON 格式标注文件。
- **核心文件：** `instances_train2017.json`。后续的多模态数据加载流水线需依赖此文件获取图像的 ground truth 描述。

### 3.3 代价评估阶梯索引 (`coco2017/indices/`)

- **内容：** 为“代价评估项目”专项定制的 10 个数据规模梯度的列表切分文件（JSONL 格式）。
- **切分逻辑：** 基于物理文件大小（Physical Size）进行精确划分，用于后续算力节点上的性能压测与并发测试。
- **文件清单（已重命名统一规范化）：**
  - `coco_physical_128M.jsonl`
  - `coco_physical_256M.jsonl`
  - `coco_physical_512M.jsonl`
  - `coco_physical_1G.jsonl` 
  - `coco_physical_1.5G.jsonl`
  - `coco_physical_2G.jsonl`
  - `coco_physical_3G.jsonl`
  - `coco_physical_4G.jsonl`
  - `coco_physical_6G.jsonl`
  - `coco_physical_8G.jsonl`

### 3.4 本地模型权重 (`coco2017/weights/`)

- **内容：** 存放图像处理过程中，特定 Data-Juicer 算子依赖的相关本地化模型组件权重文件，确保离线或内网环境下的算子能够正常调用。

#### 目录结构

```
/data-juicer/pipeline_full/
├── pipeline_2026-04-13/           # 110 条图像模板 YAML（生成器产出的原始模板）
│   ├── image_pipeline_1775733503718.yaml
│   ├── image_pipeline_1775733513924.yaml
│   └── ...（共 110 条）
├── prepare_pipeline_runs_multiscale.py   # 多尺度展开脚本
├── fix_image_tagging_null.py             # YAML 修复脚本（null 展开）
├── run_all_pipelines_with_runsdir.sh     # 批量执行 runner 脚本
├── prepared_image/                       # 展开后的运行配置
│   ├── configs/                          # 1100 个展开后的 YAML（110 模板 × 10 尺度）

│   │   └── ...

```

#### 步骤 1：模板生成

在本地 Windows 端使用 `generator_image_pipeline.py` 生成 110 条图像 pipeline 模板 YAML，上传到容器的 `pipeline_2026-04-13/` 目录。

#### 步骤 2：多尺度展开

使用 `prepare_pipeline_runs_multiscale.py` 将 110 条模板按 10 个数据规模梯度展开为 1100 条可执行配置：

```bash
python3 prepare_pipeline_runs_multiscale.py \
    --pipeline-dir ./pipeline_2026-04-25 \
    --out-dir ./prepared_image \
    --export-root /data-juicer/pipeline_full/results \
    --image-indices-dir /data/coco2017/indices \
    --modalities image
```

展开逻辑：每条模板 x 10 个尺度（128MB, 256MB, 512MB, 1G, 1.5G, 2G, 3G, 4G, 6G, 8G），生成的配置中 `dataset_path` 指向对应的 `coco_physical_*.jsonl` 索引文件。

#### 步骤 3：配置修复

展开后的配置需要修复 `image_tagging_mapper` 的参数问题（详见第 5、6 章）：

```bash
# 修复 image_tagging_mapper: null 及 text_encoder_type 路径问题
python3 fix_image_tagging_null.py ./prepared_image/configs

```

#### 步骤 4：批量执行

使用 `run_all_pipelines_with_runsdir.sh` 批量运行所有配置。该脚本逐行读取 `all_jobs.tsv`，对每条任务调用 `dj-process`，并通过 `.ok` / `.fail` 标记文件实现断点续跑：

```bash
nohup bash run_all_pipelines_with_runsdir.sh \
    /data-juicer/pipeline_full/prepared_image/manifests/all_jobs.tsv \
    /data-juicer \
    image \
    /data-juicer/pipeline_full/runs \
    > nohup_image.log 2>&1 &
```

参数说明：
| 参数 | 含义 |
|---|---|
| `$1` - manifest 路径 | `all_jobs.tsv` 任务清单 |
| `$2` - work_dir | Data-Juicer 代码根目录 `/data-juicer` |
| `$3` - modality 过滤 | `image`（只跑图像任务） |
| `$4` - runs 输出根目录 | 结果输出到 `./runs/<run_id>/` |

Runner 脚本核心逻辑：
- 读取 manifest 中每一行，提取 `config_path`、`run_name`、`export_path`
- 检查 `state/<run_name>.ok` 是否存在，存在则跳过
- 执行 `dj-process --config <yaml> --work_dir <work_dir> --export_path <export_path>`
- 成功写 `.ok`，失败写 `.fail`
- 中断后重跑时，已有 `.ok` 的任务自动跳过

#### 步骤 5：监控进度

```bash
# 实时日志

```


## 5. YAML 配置修复记录

### 5.1 `image_tagging_mapper` 参数缺失问题

**现象：** pipeline 生成器（`generator_image_pipeline.py`）在 `image_tagging_mapper` 没有随机参数时，会输出 `image_tagging_mapper: null`，即不带任何子参数。

**影响：** Data-Juicer 加载该算子时，所有参数回退到代码默认值（`pretrained_model_name_or_path="ram_plus_swin_large_14m.pth"`，`text_encoder_type="bert-base-uncased"`），在离线环境下触发 `TypeError: stat: path should be string, bytes, os.PathLike or integer, not NoneType`。

**修复方案：** 对 `prepared_image/configs/` 下 610 个受影响文件，将 `- image_tagging_mapper: null` 替换为：

```yaml
- image_tagging_mapper:
    pretrained_model_name_or_path: /data/coco2017/weights/ram_plus_swin_large_14m.pth
    text_encoder_type: /data/coco2017/weights/bert-base-uncased
    local_files_only: true
```

**修复脚本：** `fix_image_tagging_null.py`（正则替换 `- image_tagging_mapper:\s*null`）。

**对生成器的启示：** `generator_image_pipeline.py` 在生成 `image_tagging_mapper` 时，无论是否随机参数，都**必须**输出以下三个强制字段：
- `pretrained_model_name_or_path`：RAM 模型权重的本地绝对路径
- `text_encoder_type`：BERT tokenizer 的本地绝对路径
- `local_files_only: true`

### 5.2 `text_encoder_type` 值错误问题（HF 模型名 vs 本地路径）

**现象：** 部分配置文件虽然有 `text_encoder_type` 字段，但值为 `bert-base-uncased`（HuggingFace 模型 ID），而不是本地路径。

**根因：** 生成器或 sed 修复时写入了 HF 模型名而非本地绝对路径。离线模式下 `BertTokenizer.from_pretrained('bert-base-uncased')` 无法解析 vocab 文件。

**修复脚本：** `fix_image_tagging_all.py`，同时处理三种情况：
1. `image_tagging_mapper: null` → 展开为完整参数块
2. `text_encoder_type: bert-base-uncased` → 改为本地路径
3. 缺少 `text_encoder_type` → 插入

**对生成器的启示：** `text_encoder_type` 的值**必须**是本地目录绝对路径（如 `/data/coco2017/weights/bert-base-uncased`），绝对不能是 HuggingFace 模型 ID。


## 6. Data-Juicer 源码补丁记录

### 6.1 问题描述

即使 YAML 中正确配置了 `text_encoder_type: /data/coco2017/weights/bert-base-uncased`，运行时仍然报同样的 `TypeError`。

**根因：** Data-Juicer 源码的 `image_tagging_mapper.py` 和 `model_utils.py` 存在参数透传缺陷：
- `ImageTaggingMapper.__init__` 在调用 `prepare_model()` 时**硬编码**了参数，没有把 YAML 传入的 `text_encoder_type` 和 `pretrained_model_name_or_path` 透传
- `prepare_recognizeAnything_model()` 在调用 `ram.models.ram_plus()` 时**没有传递** `text_encoder_type`，导致 RAM 模型使用自己的默认值 `'bert-base-uncased'`

### 6.2 涉及文件及补丁内容

#### 文件 1：`/data-juicer/data_juicer/ops/mapper/image_tagging_mapper.py`

**原代码（第 42-44 行）：**
```python
self.model_key = prepare_model(
    model_type="recognizeAnything",
    pretrained_model_name_or_path="ram_plus_swin_large_14m.pth",
    input_size=384
)
```

**补丁后：**
```python
pretrained_model_name_or_path = kwargs.pop(
    "pretrained_model_name_or_path", "ram_plus_swin_large_14m.pth"
)
text_encoder_type = kwargs.pop("text_encoder_type", None)
kwargs.pop("local_files_only", None)
super().__init__(*args, **kwargs)
model_kwargs = dict(
    model_type="recognizeAnything",
    pretrained_model_name_or_path=pretrained_model_name_or_path,
    input_size=384,
)
if text_encoder_type:
    model_kwargs["text_encoder_type"] = text_encoder_type
self.model_key = prepare_model(**model_kwargs)
```

**要点：** 在 `super().__init__()` 之前从 `kwargs` 中 pop 出 `pretrained_model_name_or_path`、`text_encoder_type`、`local_files_only`（防止传入父类报错），然后透传给 `prepare_model()`。

#### 文件 2：`/data-juicer/data_juicer/utils/model_utils.py`

**原代码（第 688-710 行）`prepare_recognizeAnything_model` 函数：**
```python
def prepare_recognizeAnything_model(
    pretrained_model_name_or_path="ram_plus_swin_large_14m.pth",
    input_size=384, **model_params
):
    # ...
    model = ram.models.ram_plus(
        pretrained=check_model(pretrained_model_name_or_path),
        image_size=input_size, vit="swin_l"
    )
```

**补丁后：**
```python
def prepare_recognizeAnything_model(
    pretrained_model_name_or_path="ram_plus_swin_large_14m.pth",
    input_size=384, **model_params
):
    # ...
    text_encoder_type = model_params.pop("text_encoder_type", None)
    ram_kwargs = dict(image_size=input_size, vit="swin_l")
    if text_encoder_type:
        ram_kwargs["text_encoder_type"] = text_encoder_type

    model = ram.models.ram_plus(
        pretrained=check_model(pretrained_model_name_or_path),
        **ram_kwargs
    )
```

**要点：** 从 `model_params` 中提取 `text_encoder_type`，通过 `ram_kwargs` 透传给 `ram.models.ram_plus()`，使 RAM 模型初始化时使用 YAML 指定的本地 BERT 路径而非默认的 HuggingFace 模型 ID。
